# ForgePulse 设计规范

本文档定义 ForgePulse 的产品界面、交互和前端实现风格。后续新增页面、组件、图表或重构现有界面时，优先遵循本文档；如与现有代码冲突，应在保持用户体验一致的前提下逐步迁移。

## 产品定位

ForgePulse 是面向半导体测试与工业设备数据场景的平台，不是通用后台管理系统，也不是营销展示站点。

界面应传达以下气质：

- 专业、可信、工程化。
- 数据密集但不拥挤。
- 适合持续监控、快速定位异常和反复操作。
- 优先服务设备工程师、工艺工程师、数据工程师和平台运维人员。

## 设计原则

### 信息优先

页面首先回答用户正在关心的生产问题：

- 哪些设备正在运行、预警、离线？
- 哪些告警需要处理？
- 最近遥测是否异常？
- 良率、Bin、SPC 是否出现趋势变化？
- 数据来自哪里，是否可信，是否可追溯？

不要为了装饰牺牲表格密度、图表可读性或告警可发现性。

### 操作可预期

常用操作应固定在用户自然寻找的位置：

- 页面级刷新、筛选、时间范围选择放在页面顶部工具栏。
- 设备、告警、Lot、Wafer 等实体列表使用表格或紧凑列表。
- 详情页使用清晰的信息分区，先概览，再趋势，再事件或明细。
- 危险操作必须有确认、状态反馈和错误提示。

### 渐进增强

MVP 阶段优先保证端到端可用。复杂能力应逐步增加：

- 先有只读监控，再增加处理动作。
- 先有静态阈值，再增加规则配置。
- 先有基础图表，再增加联动、钻取和异常解释。
- 先有演示数据，再接入真实数据源与数据血缘说明。

## 视觉语言

### 色彩

ForgePulse 使用冷静、清晰的工业数据平台配色。颜色应服务状态识别和层级组织。

基础色：

- 页面背景：`#eef2f7`
- 主文本：`#152238`
- 次级文本：`#64748b`
- 卡片背景：`#ffffff`
- 边框：`#dbe3ef`
- 侧边栏：`#111827`

状态色：

- 正常/运行：`#15803d`
- 预警/需关注：`#ca8a04`
- 严重/告警：`#dc2626`
- 离线/未知：`#64748b`
- 分析/链接/关键指标：`#2563eb`
- 数据流/品牌强调：`#0f766e` 或 `#5eead4`

使用限制：

- 不使用大面积单一蓝紫、深蓝或渐变作为主视觉。
- 不使用装饰性光斑、漂浮球、拟物背景。
- 不用颜色作为唯一状态表达，状态必须配合文本、图标或标签。

### 字体

默认使用系统无衬线字体栈，保持当前代码中的 `Inter, ui-sans-serif, system-ui` 方案。

字号建议：

- 页面标题：20-24px
- 区块标题：16-18px
- 表格与正文：13-14px
- 辅助说明：12-13px
- 指标数字：24-32px，视卡片空间控制

不要使用随视口宽度变化的字体大小。字间距保持默认值。

### 间距与边框

- 页面内容最大宽度可约束在 `1480px` 左右，避免超宽屏信息失控。
- 主内容区留出 16-24px 内边距。
- 卡片圆角保持克制，默认不超过 8px。
- 重复列表、表格、图表卡片之间使用 16px 或 18px 间距。
- 页面不使用卡片嵌套卡片；卡片只用于独立信息模块、列表项或弹窗。

## 布局模式

### 应用壳

ForgePulse 默认使用左侧导航 + 顶部栏 + 内容区的工作台布局。

左侧导航用于稳定承载主要模块：

- 运行总览
- 设备
- 告警
- 分析
- 数据源
- 知识库
- 设置

顶部栏用于展示当前页面上下文、环境状态、最近更新时间、全局刷新和必要的快捷操作。

### 仪表盘

仪表盘应按“先状态、再异常、再分析”的顺序组织：

1. 总览指标：设备总数、运行中、预警、离线、活动告警、平均良率。
2. 设备状态与最近告警。
3. 良率、Bin、SPC 等分析图表。
4. 最新遥测、数据流状态或导入状态。

仪表盘不是报表页，不应堆满所有字段。需要更细的信息时应引导到设备详情、告警详情或分析详情页。

### 详情页

设备详情、告警详情、Lot 详情等页面应遵循：

- 顶部显示实体名称、编码、状态、最近更新时间和关键动作。
- 第一屏展示核心画像和关键指标。
- 中段展示时间序列、趋势图、SPC 或事件线。
- 底部展示明细表格、原始记录、关联告警或处理记录。

详情页可以信息密集，但每个区块必须有明确主题。

### 表格

表格用于展示可比较、可筛选、可追踪的数据：

- 设备列表、告警列表、数据源列表优先使用表格。
- 第一列应是主要实体名称，并提供详情链接。
- 状态列使用统一状态标签。
- 时间列使用统一格式化方法。
- 默认显示高价值字段，长文本放入详情页或展开区。

### 图表

图表使用 ECharts。图表应清晰表达趋势、分布或控制边界：

- 折线图用于遥测、良率和 SPC 趋势。
- 柱状图用于 Bin 分布、设备类型分布、告警数量。
- 控制图必须标注中心线、上限和下限。
- 图例、坐标轴和 tooltip 文案使用中文。
- 图表容器高度应稳定，避免加载或数据变化导致布局跳动。

## 组件规范

### 状态标签

设备状态建议：

- `running`：运行中，绿色。
- `warning`：预警，黄色。
- `offline`：离线，灰色。
- `maintenance`：维护中，蓝色或灰蓝色。

告警状态建议：

- `active`：活动中。
- `acknowledged`：已确认。
- `resolved`：已恢复。

告警等级建议：

- `info`：信息。
- `warning`：警告。
- `critical`：严重。

状态标签应集中封装，避免在页面中重复硬编码颜色和文案。

### 按钮与图标

按钮应表达明确动作：

- 刷新使用刷新图标。
- 新增使用加号图标。
- 导出使用下载图标。
- 查看详情使用链接或箭头图标。
- 危险操作使用红色危险按钮，并配合确认。

图标优先使用项目现有图标库。文字按钮只用于明确命令，不用于装饰。

### 空状态、加载和错误

所有远程数据区块都需要考虑：

- 首次加载。
- 局部刷新。
- 空数据。
- API 错误。
- 演示模式数据。

错误提示应说明“发生了什么”和“用户可以怎么检查”，例如提示确认 `platform-api` 是否启动。

## 文案规范

界面文案默认使用简体中文。

写法要求：

- 简短、明确、面向任务。
- 保留必要英文技术名词，如 MQTT、Kafka、SPC、Lot、Wafer、API。
- 不在页面上解释显而易见的功能。
- 不把开发说明、快捷键说明或实现细节放进正式界面。

示例：

- 使用“最近心跳”，不要使用“最后一次 heartbeat 时间戳”。
- 使用“活动告警”，不要使用“当前未关闭的 alarm entities”。
- 使用“数据加载失败”，不要使用“请求异常”作为唯一提示。

## 前端实现约定

当前前端技术栈：

- React
- TypeScript
- Vite
- Ant Design
- ECharts
- React Router

实现要求：

- 页面级业务代码放入 `web/src/features/*`。
- 通用图表、状态标签、格式化方法和 API 客户端放入 `web/src/shared/*`。
- 应用壳、路由和导航放入 `web/src/app/*`。
- 新增页面优先复用现有布局、状态标签、图表封装和格式化工具。
- 不为单个页面引入新的大型 UI 库。
- CSS 变更应尽量集中在 `web/src/styles.css` 或局部组件样式中，避免分散重复。

## 响应式规则

ForgePulse 主要服务桌面端和大屏工作站，但必须保证移动端可读：

- 桌面端优先展示表格和多列图表。
- 中等屏幕改为两列或单列。
- 移动端工具栏、表格和图表应纵向堆叠。
- 表格在窄屏可横向滚动，不能让文本重叠。
- 图表高度使用稳定值，移动端可适当降低。

## AI 协作提示

当 AI 代理为 ForgePulse 修改前端时，应先读取：

1. `AGENTS.md`
2. `DESIGN.md`
3. `docs/progress.md`
4. 相关页面和共享组件代码

生成或修改页面时，应优先保持以下一致性：

- 工业数据平台定位。
- 简体中文文案。
- Ant Design + ECharts 技术路线。
- 左侧导航、顶部栏、内容区布局。
- 状态色、标签文案和图表风格。
- 数据密集但克制的界面组织。

不要把 ForgePulse 设计成营销落地页、品牌展示页、消费级 App 或纯视觉 demo。用户打开应用后，应直接进入可操作、可监控、可分析的工作界面。
